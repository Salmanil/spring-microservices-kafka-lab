# Kafka Doubts And Answers

## 1. How do I know which consumer has which partition? Did I set it anywhere?

You did not manually assign partitions in code.

Kafka assigns partitions automatically inside a consumer group.

In this project, the two listeners in `notification-service` are in the same group, so Kafka distributes partitions between them.

Check it with:

```powershell
docker exec kafka /opt/kafka/bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 --describe --group notification-group
```

What you set:

- the consumer group

What Kafka sets:

- partition ownership

## 2. When the DB is inserted, does it go through Kafka first or after DB create?

In the current code, DB happens first, then Kafka event publishing happens after.

Flow:

1. save in DB
2. publish Kafka event

So Kafka is acting like a change event after the database write.

## 3. Why does Kibana overwrite instead of adding a new log for POST/PUT?

Because the Elasticsearch sink is using the Kafka message key as the Elasticsearch document id.

That means:

- same key -> same document id
- next event with same key -> overwrite existing document

So Kibana is showing the latest document state for that key, not a new appended document every time.

If you want append-only event logs:

- use autogenerated ids
- or use a unique event id in the key
- or configure the sink differently

## 4. Why is timestamp not showing in Kibana?

`kafka_timestamp` exists, but it is currently stored like a number and not necessarily mapped as a Kibana time field.

So Kibana may not automatically use it for time filtering like `@timestamp`.

You can still see the value, but for proper time-based usage it should be mapped as a date field.

## 5. Why did this ksqlDB query show only headers and no rows?

```powershell
docker exec -i ksqldb-cli ksql http://ksqldb-server:8088 -e "SELECT EMPID, NAME, DEPTID, SALARY FROM EMPLOYEE_EVENTS_JSON EMIT CHANGES LIMIT 5;"
```

Because `EMIT CHANGES` usually waits for new incoming records.

If no new messages arrive after the query starts, you may see only the headers.

Use:

```powershell
docker exec -i ksqldb-cli ksql http://ksqldb-server:8088 -e "SET 'auto.offset.reset'='earliest'; SELECT EMPID, NAME, DEPTID, SALARY FROM EMPLOYEE_EVENTS_JSON EMIT CHANGES LIMIT 5;"
```

Or:

```powershell
docker exec -i ksqldb-cli ksql http://ksqldb-server:8088 -e "PRINT 'employee-events' FROM BEGINNING LIMIT 5;"
```

## 6. Does Schema Registry ensure message format? If I send `hello`, will it go to Elasticsearch?

In the current project, Schema Registry is not actively enforcing your app payload format.

Why:

- producer uses string serializer
- payloads are plain strings containing JSON text
- not Avro / Protobuf / JSON Schema serializer flow

So:

- Kafka can accept `"hello"`
- consumer can consume `"hello"`
- Elasticsearch sink may fail because it expects JSON

Schema Registry becomes truly useful when producers and consumers use registry-aware serializers.

## 7. If I post the same message again and again, Kafka still produces and consumes it. Isn’t that wrong?

No. That is normal Kafka behavior.

Kafka stores records, not business uniqueness rules.

If the same message is sent 3 times:

- Kafka stores 3 records
- consumers read 3 records

Kafka does not automatically deduplicate your business messages.

If Kibana did not show 3 separate documents, that is because Elasticsearch overwrote them using the same key.

## 8. Kafka Connect fails when Kafka is stopped and started. What should I do?

This is common in local lab setups.

Recommended steps:

1. wait for Kafka to fully come back
2. check connector status

```powershell
Invoke-WebRequest http://localhost:8083/connectors/employee-events-elasticsearch-sink/status -UseBasicParsing
```

3. if task failed, restart connector

```powershell
Invoke-WebRequest -Method Post http://localhost:8083/connectors/employee-events-elasticsearch-sink/restart -UseBasicParsing
```

4. if Connect itself is unhealthy, restart the container

```powershell
cd C:\Docker_compose\kafka-setup
docker compose restart kafka-connect
```

## 9. If Kafka Connect is down and I send POST, it still produced and consumed. Where did it get consumed?

It was consumed by `notification-service`, not by Elasticsearch.

Important distinction:

- `notification-service` consumes Kafka messages directly
- Kafka Connect also consumes Kafka messages independently
- Elasticsearch only receives messages through Kafka Connect

So if Kafka Connect is down:

- producer still writes to Kafka
- notification-service still consumes
- Elasticsearch does not receive new records

## Simple Flow Summary

1. DB save happens first
2. Kafka event is published after DB save
3. `notification-service` consumes from Kafka
4. Kafka Connect also consumes from Kafka
5. Elasticsearch receives data only from Kafka Connect
6. Kibana shows only what Elasticsearch has
